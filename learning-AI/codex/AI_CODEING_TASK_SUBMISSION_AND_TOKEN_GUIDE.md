# AI_CODEING 任务提交机制与 Token 使用建议

## 1. 文档目的

本文档用于说明在当前 Codex 环境下，用户提交一个 AI 编程任务后，任务是如何逐步组织并发送给 LLM 后台服务的，以及用户应如何控制上下文、token 和任务颗粒度，提升代码生成质量与稳定性。

本文档既适用于当前项目，也适用于大多数代码仓库。

## 2. 先说结论

用户给 Codex 提任务时，不是“只发送你输入的一句话”。

实际提交给 LLM 后台的内容，通常会由以下几类信息共同组成：

1. 用户当前这次输入的任务描述。
2. 当前会话的历史对话。
3. 系统级约束。
4. 开发者级约束。
5. 仓库中的 `AGENTS.md`、工作区规则、技能说明等本地指令。
6. Codex 为完成任务主动读取到的代码、目录结构、日志、配置文件、命令输出。
7. Codex 在执行过程中产出的中间分析、工具结果和修改后的代码上下文。

这意味着：

1. 用户输入越清晰，模型越容易对准目标。
2. 仓库内杂乱、重复、冲突的文档越多，模型越容易被干扰。
3. token 不只是“你输入的字数”，而是整轮上下文的总和。
4. 真正影响效果的，往往是“上下文组织质量”，而不只是“模型本身”。

## 3. 当前 Codex 下的任务提交链路

说明：

下面描述的是当前 Codex 工作方式下，一个任务通常如何组织上下文并提交到 LLM 后台。

这里的“拉取”“扫描”“读取”分三种情况：

1. 必然进入上下文的内容。
2. 高概率会先被枚举或检查的内容。
3. 只有与任务相关时才会按需读取的内容。

因此，不能把它理解成“每次都把整个仓库完整上传一遍”，更接近真实情况的是：

1. 平台先带上固定规则。
2. Codex 再按任务需要，逐步查看目录、读取文件、运行命令。
3. 被读取到的结果再分批回流给 LLM。

## 3.1 用户视角的表象

用户看到的是：

1. 提交一句任务。
2. Codex 读取文件、执行命令、分析代码。
3. Codex 修改代码、生成文档、编译或测试。
4. Codex 返回结果。

## 3.2 更接近真实的内部链路

从工程视角，通常可以理解为下面这个过程：

1. 用户输入任务。
2. 平台把本轮用户消息与已有对话历史拼接。
3. 平台附加系统提示词、开发者提示词、本地仓库规则。
4. 模型先做任务理解与计划。
5. 模型决定是否需要读文件、查目录、运行命令。
6. 工具执行结果再被送回模型。
7. 模型基于“用户需求 + 仓库上下文 + 工具结果”继续推理。
8. 如果需要改代码，模型生成补丁或命令。
9. 修改结果、测试结果继续回流模型。
10. 模型最终给出可交付结果。

## 3.3 在当前环境里，哪些内容特别容易进入上下文

结合当前仓库，以下信息通常会显著影响提交质量：

1. 根目录或子目录中的 `AGENTS.md`。
2. `app.bak` 下已有的大量分析文档和说明文档。
3. 与任务相关的源码文件、配置文件、日志文件。
4. 构建方式、测试方式、部署方式。
5. 串口、板端、网页登录方式等环境说明。

所以，仓库里每一份“规范文档”其实都在潜在地影响 Codex 的判断。

更详细的“全流程拉取与扫描说明”，见：

## 4. 什么是 Token

## 4.1 简单理解

token 可以理解为模型处理文本时使用的“最小计费与计算单元”。

它不是严格等于：

1. 一个汉字。
2. 一个英文单词。
3. 一个字符。

真实情况更像是：文本会被切分成若干片段，每个片段就是 token。

## 4.2 为什么要关心 Token

因为每次请求，模型能处理的上下文是有限的。

会话里放入的内容越多：

1. 成本越高。
2. 延迟越高。
3. 重点越容易被稀释。
4. 模型越可能忽略关键约束。
5. 越容易出现“前面说过，后面忘了”的现象。

## 4.3 在 AI 编程里，Token 的主要消耗来源

1. 用户任务描述过长但没有结构。
2. 粘贴整份日志、整份源码、整份接口文档。
3. 历史对话过长，没有阶段性收敛。
4. 仓库里存在大量相似但冲突的说明文档。
5. 一次任务同时要求分析、重构、修复、测试、写文档、部署。
6. 编译输出和运行日志过大。

## 4.4 一个实用判断标准

如果你一句任务里同时包含下面 4 类以上目标，通常就应该拆任务：

1. 问题分析。
2. 方案设计。
3. 代码实现。
4. 联调测试。
5. 文档整理。
6. 部署发布。

## 5. 用户提交任务时，后台到底“提交了什么”

从提交内容构成来看，通常包括以下几层：

## 5.1 指令层

这部分决定模型“应该怎么工作”：

1. 系统规则。
2. 开发者规则。
3. 仓库规则。
4. 当前目录内 `AGENTS.md` 约束。

例如：

1. 要不要先看代码再动手。
2. 是否必须用某种编辑方式。
3. 是否优先做代码改动而不是只给建议。
4. 是否必须先发进度说明。
5. 是否限制联网。

## 5.2 任务层

这部分决定模型“要做什么”：

1. 用户当前这次的明确需求。
2. 用户附加的交付要求。
3. 输出路径要求。
4. 是否要求编译、测试、生成文档。

## 5.3 上下文层

这部分决定模型“基于什么信息做判断”：

1. 目录结构。
2. 相关源码。
3. 相关日志。
4. 配置文件。
5. 历史修改结果。
6. 既有 README、设计文档、分析报告。

## 5.4 工具回流层

这部分决定模型“读到了什么真实结果”：

1. `ls`、`find`、`sed`、`git` 输出。
2. 编译输出。
3. 测试输出。
4. 错误栈。
5. 运行日志。
6. 代码 diff。

这也是为什么“让 Codex 自己先看代码”通常比“用户手工复制一大段代码”更省 token。

## 6. 用户提交任务的全面建议

## 6.1 提需求时，优先写清楚这 8 件事

建议按下面结构组织：

1. 目标是什么。
2. 问题现象是什么。
3. 涉及哪个模块或文件。
4. 期望修改到什么程度。
5. 是否允许改接口、数据结构、配置。
6. 是否需要编译、测试、部署。
7. 交付物放在哪里。
8. 有哪些明确不能动的部分。

推荐写法示例：

```text
目标：修复 IPC Web 页面上传配置后偶发崩溃的问题。
现象：点击“导入参数”后，服务端偶发段错误。
范围：优先检查 app.bak/component/webserver 和 config 相关代码。
要求：先定位根因，再直接改代码，并补一份简短分析文档到 app.bak/z-readme。
限制：不要改前端接口字段；不要删除现有日志；完成后给出编译验证方式。
```

## 6.2 一次只让模型完成一个主目标

不推荐：

```text
帮我分析问题、重构代码、优化前端、统一接口、补齐测试、整理所有文档并顺便部署到板端。
```

推荐拆成多轮：

1. 先定位问题和根因。
2. 再修改代码。
3. 再补测试和文档。
4. 最后做部署和板端验证。

## 6.3 提供“定位信息”比提供“大量原文”更有效

优先给：

1. 文件路径。
2. 函数名。
3. 错误现象。
4. 复现步骤。
5. 关键日志关键词。

少直接给：

1. 整个 5000 行文件。
2. 整份编译日志。
3. 大段重复聊天背景。

## 6.4 用“验收标准”约束结果

如果没有验收标准，模型很容易只做到“看起来改了”，但不一定真正满足需求。

建议明确写：

1. 成功条件。
2. 失败条件。
3. 验证步骤。
4. 输出文件位置。

例如：

```text
验收标准：
1. 能正常编译。
2. 导入配置后不崩溃。
3. 页面返回成功提示。
4. 生成一份问题分析文档到 app.bak/z-readme。
```

## 6.5 约束“不要做什么”

这类信息很重要，往往能显著减少误改：

1. 不要改数据库结构。
2. 不要改对外 API。
3. 不要删历史兼容逻辑。
4. 不要引入新三方库。
5. 不要改串口登录流程。

## 6.6 当任务很大时，要求阶段性交付

建议明确要求：

1. 先输出分析。
2. 再动代码。
3. 再编译验证。
4. 再写文档。

这种方式可以降低大任务一次性失控的概率。

## 7. 面向所有项目的通用规范建议

## 7.1 仓库规范

建议每个项目至少具备以下文档：

1. `AGENTS.md`：告诉 Codex 如何工作。
2. `README.md`：告诉人类如何构建、运行、测试。
3. `docs/ARCHITECTURE.md`：核心模块关系。
4. `docs/DEBUG_GUIDE.md`：常见故障排查路径。
5. `docs/CHANGE_RULES.md`：哪些模块可改，哪些模块谨慎改。

## 7.2 日志规范

建议统一日志输出规则：

1. 每条错误日志要带模块名。
2. 关键流程要带输入参数摘要。
3. 异常要带返回码和错误原因。
4. 不要大量输出无意义重复日志。

原因很简单：日志越可检索，Codex 越容易快速定位问题。

## 7.3 目录规范

建议把以下内容分开：

1. 源码。
2. 文档。
3. 原型。
4. 样例。
5. 临时脚本。
6. 调试日志。

不要把：

1. 临时草稿。
2. 旧方案。
3. 重复说明。
4. 已失效文档。

长期混在核心路径下，否则会污染上下文判断。

## 7.4 命名规范

建议文档名直接表达用途：

1. `BUG_xxx_ROOT_CAUSE.md`
2. `FEATURE_xxx_IMPLEMENTATION.md`
3. `MODULE_xxx_ANALYSIS.md`
4. `BUILD_AND_TEST_GUIDE.md`

比起：

1. `说明.md`
2. `最终版.md`
3. `新方案2.md`
4. `修复最终最终版.md`

更适合长期协作，也更适合 AI 检索。

## 7.5 任务规范

推荐每次任务都尽量包含：

1. 背景。
2. 目标。
3. 范围。
4. 约束。
5. 验收。
6. 输出位置。

这 6 项足以覆盖大多数 AI 编程协作场景。

## 8. 针对当前项目的具体建议

## 8.1 当前项目的优势

当前项目已经具备一定基础：

1. 主项目路径明确。
2. 编译方式明确。
3. 测试入口明确。
4. 板端登录方式明确。
5. 已有较多专项分析文档。

## 8.2 当前项目的风险点

从 AI 协作角度看，当前项目也有几个典型风险：

1. `z-readme` 下文档较多，主题重复度偏高。
2. 一些文档命名不统一。
3. 同类问题可能存在多份“最终版”说明，容易造成上下文冲突。
4. 如果把过多历史分析文档都放进同一目录，AI 检索时可能读到过期结论。

## 8.3 建议你后续补充的治理动作

建议逐步做下面几件事：

1. 给 `z-readme` 按主题分子目录。
2. 对历史文档加状态标记：`active`、`archived`、`obsolete`。
3. 为关键模块建立统一入口文档。
4. 为编译、部署、串口登录建立固定模板。
5. 对“最终版”“修复版”“V2/V3/V4”文档做清理归并。

## 9. 推荐的提问模板

## 9.1 缺陷修复

```text
任务：修复一个问题。
现象：描述复现步骤和报错。
范围：给出目录、文件或模块。
限制：哪些地方不能改。
要求：直接改代码，必要时补日志和文档。
验收：编译通过，问题复现消失，给出验证步骤。
```

## 9.2 新功能开发

```text
任务：实现一个新功能。
背景：为什么做。
输入输出：接口、参数、页面或配置项。
范围：建议优先修改哪些模块。
限制：兼容旧逻辑，不改外部协议。
验收：功能可用，旧功能不回退，补文档。
```

## 9.3 分析类任务

```text
任务：只分析，不要先改代码。
目标：解释调用链、数据流、提交机制或根因。
范围：限定要看的目录。
输出：写成文档到指定路径。
要求：结论、证据、风险、建议分别列清楚。
```

## 10. 关于 Token 的最终建议

如果希望 AI 编程效果稳定，最重要的不是“把所有信息一次性都喂进去”，而是：

1. 只给当前任务必要信息。
2. 用结构化方式描述需求。
3. 把大任务拆成多轮小任务。
4. 给出明确范围和禁区。
5. 要求可验证的验收结果。
6. 定期整理仓库文档，避免过期信息污染上下文。

## 11. 一句话原则

高质量 AI 编程提交，不是“信息越多越好”，而是“目标明确、范围收敛、约束清楚、验证可执行”。
