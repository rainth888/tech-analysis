# AI_CODEING 每次提交任务时的上下文拉取全流程

## 1. 文档目的

本文档专门回答一个问题：

在当前 Codex 环境下，用户每提交一次任务，Codex 和其背后的 LLM 服务，通常会经历什么流程，自动拉取哪些信息，扫描哪些目录，读取哪些文件类型。

本文档强调两点：

1. 这里描述的是“高概率、典型、可操作”的真实工作流，不是平台源码级的绝对内部实现说明。
2. 为了帮助你规范项目，我会把“当前项目常见路径”和“跨项目普遍会被检查的路径”都列出来。

## 2. 先说结论

一次任务提交，并不是“把你这句话单独发给模型”。

更接近真实情况的是：

1. 先带上固定指令集。
2. 再带上当前对话历史。
3. 再看当前工作目录和仓库规则。
4. 再根据任务需要，逐步扫描目录、读取文件、运行命令。
5. 再把这些结果分批送回 LLM。
6. 如果继续分析、修改、测试，新的结果还会反复继续回流。

所以，一个任务的真实输入，通常由下面 5 类信息叠加而成：

1. 指令。
2. 对话。
3. 仓库规则。
4. 文件与命令结果。
5. 中间推理后的后续工具结果。

## 3. 一次任务的完整阶段流程

## 3.1 阶段 0：固定系统上下文装载

在用户任务真正开始之前，模型通常已经带着一批固定指令工作。这部分一般包括：

1. 系统级规则。
2. 开发者级规则。
3. 工具使用规则。
4. 当前运行环境信息。
5. 工作目录信息。
6. 权限限制。

在当前环境中，这部分至少包括：

1. 当前日期、时区。
2. 当前工作目录。
3. shell 类型。
4. 可用工具列表。
5. 文件编辑约束。
6. 是否允许联网。
7. 是否允许提权执行命令。

这部分内容属于“必然进入上下文”的内容。

## 3.2 阶段 1：会话上下文拼接

用户提交任务后，平台会把当前这轮消息与已有会话历史进行拼接。

高概率包含：

1. 用户这次的原始任务文本。
2. 之前几轮对话。
3. 上一轮工具执行结果摘要。
4. 上一轮修改结果摘要。

如果会话很长，平台不一定保留原样全文，但会尽量保留对当前任务有帮助的历史上下文。

## 3.3 阶段 2：工作区规则定位

这一步是 AI 编程场景里非常关键的一步。

Codex 往往会优先检查当前工作区中会影响执行方式的规则文件。

高概率会关注：

1. 当前目录相关的 `AGENTS.md`。
2. 上层目录中的 `AGENTS.md`。
3. 当前任务明确提到的本地规范文档。
4. 技能目录中的 `SKILL.md`。

在当前仓库中，实际已经明确存在并影响行为的内容包括：

1. 仓库相关 `AGENTS.md`。
2. 你在任务中附带的本地编译、测试、板端说明。
3. 可用技能目录说明。

这部分内容通常会优先于普通源码被考虑，因为它决定“怎么做”，而不是“做什么”。

## 3.4 阶段 3：目录级初步扫描

在理解任务后，Codex 通常会先做一次低成本的目录扫描，而不是直接读很多文件正文。

典型动作包括：

1. 查看当前路径。
2. 列目录。
3. 查找目标文件。
4. 枚举相关子目录。
5. 搜索文件名或关键字。

这一阶段高概率读取到的信息是：

1. 顶层目录名。
2. 子目录列表。
3. 可疑模块路径。
4. 文档目录位置。
5. 日志目录位置。
6. 构建脚本位置。

这一阶段更像“地图构建”，不等于已经把所有文件内容都拉进来了。

## 3.5 阶段 4：按任务相关性读取具体文件

目录扫描完成后，Codex 才会按任务需要读取文件正文。

典型优先级一般是：

1. 规则文件。
2. README / 构建说明。
3. 任务直接点名的文件。
4. 与报错最相关的源码。
5. 与复现最相关的日志。
6. 与验证最相关的脚本或配置。

读取方式通常是：

1. 只读文件开头。
2. 只读相关段落。
3. 搜索命中后再局部展开。
4. 只读取关键函数附近行号。

这也是为什么“文件虽然在仓库里，但不一定每次都被完整送入模型”。

## 3.6 阶段 5：工具结果回流到 LLM

当 Codex 运行命令、查看文件、搜索文本后，得到的是工具输出。

这些输出会成为新一轮模型输入的一部分。

常见回流内容包括：

1. `pwd` 结果。
2. `find` / `ls` 结果。
3. `sed` 读取的源码片段。
4. `git diff` 或文件变更。
5. 编译输出。
6. 测试输出。
7. 错误日志。

模型会基于这些真实结果继续决定下一步。

## 3.7 阶段 6：修改、验证、再回流

如果任务需要改代码，那么流程会继续循环：

1. 生成修改计划。
2. 修改文件。
3. 读取修改后的结果。
4. 运行编译或测试。
5. 再把编译或测试结果送回 LLM。
6. 再决定是否补改。

所以，一次任务可能不是一次提交，而是多轮“读取 -> 推理 -> 执行 -> 回流”的连续过程。

## 4. 每次任务通常会自动拉取哪些内容

## 4.1 必然会进入上下文的内容

这部分几乎每次都会出现：

1. 用户本轮任务。
2. 最近几轮对话历史。
3. 系统规则。
4. 开发者规则。
5. 当前环境信息。
6. 当前工作目录信息。
7. 工具使用规则。

## 4.2 高概率会被检查的本地规则文件

如果存在，通常优先级很高：

1. `AGENTS.md`
2. `README.md`
3. `readme.md`
4. `_readme-baseinfo.md`
5. `SKILL.md`
6. 构建与测试说明文档
7. 与当前目录紧邻的任务规范文档

## 4.3 高概率会被查看的目录类型

无论具体项目是什么，下面这些目录类型都很容易被优先扫描：

1. 源码目录：`src`、`app`、`component`、`modules`、`lib`
2. 前端目录：`web`、`webapp`、`ui`、`frontend`
3. 服务端目录：`server`、`backend`、`service`
4. 配置目录：`config`、`configs`
5. 文档目录：`docs`、`z-readme`
6. 脚本目录：`scripts`、`tools`
7. 测试目录：`test`、`tests`
8. 日志目录：`logs`
9. 构建目录：`build`、`dist`
10. 示例目录：`samples`、`examples`

## 4.4 在当前项目里，高概率会被扫描的路径

结合当前仓库结构，后续任务里高概率会被优先关注的目录包括：

1. `app.bak`
2. `app.bak/component`
3. `app.bak/component/webapp`
4. `app.bak/component/webserver`
5. `app.bak/component/config`
6. `app.bak/component/ai_app`
7. `app.bak/scripts`
8. `app.bak/docs`
9. `app.bak/logs`
10. `app.bak/z-readme`
11. `app.bak/z-samples`
12. `app.bak/z-prototype`
13. `app.bak/.agent`

这些目录不代表每次全部都会深读，但很容易先被枚举或局部读取。

为了便于你给所有项目都做规范，下面这些目录名在其他项目里也经常是 AI 首先去找的地方：

1. `src`
2. `include`
3. `cmd`
4. `pkg`
5. `internal`
6. `api`
7. `proto`
8. `deploy`
9. `docker`
10. `.github`
11. `.gitlab`
12. `ci`
13. `ops`
14. `infra`
15. `firmware`
16. `board`
17. `device`
18. `kernel`
19. `drivers`
20. `third_party`
21. `vendor`
22. `packages`
23. `monorepo` 下的各子工程目录

原因不是这些目录一定会被全部读取，而是它们往往代表：

1. 入口代码。
2. 公共库。
3. 接口定义。
4. 部署方式。
5. CI 规则。
6. 板级或驱动侧逻辑。

## 6. 平台会不会把整个项目都自动上传

通常不应该这样理解。

更准确的理解是：

1. 规则类上下文和会话历史，是直接附带的。
2. 仓库内容更多是“按需枚举、按需读取、分段回流”。
3. 越相关的文件，越容易被读取。
4. 越大的文件，越可能只读片段。
5. 越不相关的目录，越可能只看到名字而不看正文。

所以，项目治理重点不是“假设全部文件每次都被完整上传”，而是：

1. 假设任何显眼的规则文档都有机会影响模型。
2. 假设任何与任务强相关的目录都会被快速定位。
3. 假设命名混乱和重复文档会显著干扰结果。

## 7. 一个真实可执行的“拉取顺序”参考

你可以把每次任务大致理解为下面顺序：

1. 装载系统和开发者规则。
2. 装载环境信息和当前工作目录。
3. 装载用户当前任务和最近会话历史。
4. 检查 `AGENTS.md`、本地规范、技能说明。
5. 看顶层目录和相关子目录。
6. 找目标模块、目标文件、目标日志。
7. 读取少量关键文件片段。
8. 如果信息不够，再继续展开更多文件。
9. 需要执行时，运行命令并读取输出。
10. 需要修改时，生成补丁并再次读取修改结果。
11. 需要验证时，运行编译/测试并读取结果。
12. 最终总结并输出交付物。

## 8. 为了让 Codex 更稳定，你应该怎样整理项目

## 8.1 优先整理规则文件

建议确保：

1. `AGENTS.md` 简短、稳定、无冲突。
2. 编译说明只有一份主入口。
3. 测试说明只有一份主入口。
4. 板端登录方式固定写法。

## 8.2 优先整理文档目录

建议：

1. 正式文档和临时草稿分开。
2. 历史文档归档。
3. 同一主题只保留一份主说明。
4. 文件名带统一前缀。

## 8.3 优先整理日志和样例

建议：

1. 大日志不要长期堆在核心目录。
2. 样例和原型单独放目录。
3. 可复现问题要保留最小日志样本。

## 9. 对你当前仓库的直接建议

为了后续更适合 AI 协作，建议你逐步形成下面的固定结构：

1. `app.bak/AGENTS.md` 或根级规则文件作为唯一主规则入口。
2. `app.bak/z-readme/AI_CODEING_*.md` 作为 AI 协作文档统一前缀。
3. `app.bak/z-readme/archive/` 用于放过期文档。
4. `app.bak/logs/` 只保留近期关键日志。
5. `app.bak/docs/` 只放正式说明文档。

## 10. 一句话原则

每次提交任务时，Codex 不是一次性“全仓上传”，而是“先装载固定规则，再围绕任务逐步扫描、局部读取、持续回流”；所以你最该优化的是规则文件、目录命名、文档去重和任务描述结构。
