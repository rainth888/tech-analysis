# AI_CODEING 协作 SOP

## 1. 目标

本文档用于规范用户与 Codex 的协作方式，减少无效来回、误改和上下文污染，适用于大部分代码项目。

## 2. 标准协作流程

## 2.1 小任务

适合：

1. 单点 bug 修复。
2. 单文件修改。
3. 补一份短文档。
4. 定位一个明确报错。

建议流程：

1. 用户给出目标、范围、限制、验收。
2. Codex 先读取相关代码。
3. Codex 直接修改。
4. Codex 运行最小必要验证。
5. Codex 输出结果。

## 2.2 大任务

适合：

1. 模块重构。
2. 新功能开发。
3. 多模块联调。
4. 涉及板端、网页、服务端协同的问题。

建议流程：

1. 第一轮只做分析和计划。
2. 第二轮做核心代码修改。
3. 第三轮做编译和测试。
4. 第四轮补文档和收尾。

## 3. 用户侧最佳实践

## 3.1 提任务时必须说清楚

1. 你要结果，不只是建议，还是只要分析。
2. 是否允许直接改代码。
3. 是否要求编译、运行、测试。
4. 输出文档放在哪里。
5. 哪些区域不能动。

## 3.2 用户应尽量避免

1. 一句话塞进多个目标。
2. 把大量无筛选日志直接粘给模型。
3. 同一个问题连续切换目标。
4. 让模型猜路径、猜模块、猜验收标准。

## 3.3 最好提供的信息

1. 错误截图对应的文字描述。
2. 文件路径。
3. 函数名。
4. 接口 URL。
5. 关键报错关键词。
6. 复现步骤。

## 4. Codex 侧工作原则

在当前环境里，一个相对稳定的执行原则通常是：

1. 先读代码，再下结论。
2. 先缩小范围，再修改。
3. 优先最小必要改动。
4. 变更后尽量做验证。
5. 输出结论时区分事实、推断、风险。

## 5. 文档治理建议

为了让 AI 长期稳定工作，建议项目维护以下规则：

1. 每个问题只保留一份主结论文档。
2. 历史文档放归档目录。
3. 文档名必须可检索、可判断主题。
4. 过期结论必须标注。
5. 模块入口文档要短，专题文档可详细。

## 6. Token 控制 SOP

建议采用下面做法：

1. 先让 Codex 自己读文件，不要先粘贴整份源码。
2. 先给关键词和路径，再补必要原文。
3. 长日志先摘要，再按需看原文。
4. 长任务拆轮次。
5. 阶段性结束后重新开一个干净任务，避免历史上下文持续膨胀。

## 7. 面向当前项目的落地建议

针对当前仓库，建议以后提交任务时优先写成下面形式：

```text
任务：xxx
范围：app.bak/component/xxx、app.bak/component/yyy
背景：xxx
要求：先分析 / 直接改代码 / 编译验证 / 写文档
限制：不要改 xxx
输出：app.bak/z-readme/xxx.md
验收：xxx
```

## 8. 收尾要求

一次任务完成后，建议固定要求 Codex 输出：

1. 改了什么。
2. 为什么这样改。
3. 是否完成验证。
4. 还有哪些风险未覆盖。

这样后续接手的人类或下一轮 AI，才能低成本继续工作。
