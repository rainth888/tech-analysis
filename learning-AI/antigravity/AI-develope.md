# AI 代理开发机制详解 — Gemini Code Assist / Antigravity

## 一、核心概念

### 1.1 系统提示词（System Prompt）

系统提示词是**每次对话开始时，平台自动注入给 LLM 的一大段指令文本**。用户看不到这段内容，但它决定了 AI 的行为模式。

**包含的内容：**

| 内容             | 说明                                 | 是否可控     |
|------------------|--------------------------------------|--------------| 
| 身份定义         | AI 的角色、行为规范                  | ❌ 不可控    |
| 工具定义         | 所有可用工具的 JSON Schema           | ❌ 不可控    |
| 技能（Skills）   | 扩展能力的指令集                     | ❌ 不可控    |
| 工作流索引       | `.agents/workflows/` 中的文件列表    | ✅ 可控      |
| 用户自定义规则   | 用户设置的全局行为规则               | ✅ 可控      |
| 知识库摘要（KI） | 过往对话中提炼的知识条目摘要         | △ 间接影响   |
| 对话历史摘要     | 最近 20 条对话的标题和摘要           | △ 间接影响   |
| 用户当前状态     | 打开的文件、光标位置、浏览器状态     | △ 自动采集   |

**Token 开销：** 系统提示词通常消耗 **数千到上万个 token**，是每次对话的固定开销。

---

### 1.2 工具定义（Tool Definitions）

工具是 AI 与外部世界交互的接口。每个工具以 JSON Schema 的形式定义在系统提示词中。

**当前可用的工具类别：**

| 类别     | 工具示例                                              | 用途             |
|----------|-------------------------------------------------------|------------------|
| 文件操作 | `view_file`, `write_to_file`, `replace_file_content`  | 读写文件         |
| 搜索     | `grep_search`, `find_by_name`                         | 搜索代码和文件   |
| 终端     | `run_command`, `command_status`                       | 执行命令行操作   |
| 浏览器   | `browser_subagent`                                    | 浏览器自动化操作 |
| 网络     | `read_url_content`, `search_web`                      | 网页读取和搜索   |
| 图像     | `generate_image`                                      | AI 图像生成      |
| 通信     | `notify_user`                                         | 与用户交互       |
| 任务管理 | `task_boundary`                                       | 任务进度追踪     |

**Token 开销：** 每个工具的完整 JSON Schema 定义约 100-500 token，所有工具加起来约 **3000-5000 token**（固定开销，无法减少）。

---

### 1.3 工作流（Workflows）

#### 发现机制

系统在每次对话开始时，自动扫描项目中以下目录：

```
.agents/workflows/
.agent/workflows/
_agents/workflows/
_agent/workflows/
```

#### 两阶段加载

```
阶段 1（每次对话）：只读取 YAML frontmatter 中的 description
                    → 注入系统提示词，作为索引（~30-80 token/个）

阶段 2（触发时）：  通过 view_file 读取 .md 文件的完整正文
                    → 按步骤执行（~200-800 token/个）
```

#### 工作流文件结构

```markdown
---
description: 触发条件描述（此行每次对话都加载）
---

# 正文：具体执行步骤（仅触发时加载）
1. 步骤一
2. 步骤二
```

#### 触发方式

| 方式           | 示例                  | 说明                                 |
|----------------|-----------------------|--------------------------------------|
| 斜杠命令       | `/save-progress`      | 文件名直接匹配                       |
| 自然语言匹配   | "保存进度"、"我要走了" | AI 根据 description 中的关键词判断   |

#### 特殊注解

```
// turbo       → 该步骤自动执行，不需用户确认
// turbo-all   → 所有步骤自动执行
```

---

### 1.4 项目规则文件（AGENTS.md）

`AGENTS.md` 是放在项目根目录中的**项目级规则文件**，用于定义 AI 在当前项目中应遵循的固定规则。

#### 加载机制

**`AGENTS.md` 不是每次对话自动注入系统提示词的**，而是 AI 在工作时主动读取的。

| 对比             | 加载时机           | 固定 Token 开销 |
|------------------|--------------------|-----------------|
| 系统提示词       | 每次对话自动注入   | ✅ 有           |
| User Rules       | 每次对话自动注入   | ✅ 有           |
| 工作流索引       | 每次对话自动注入   | ✅ 有           |
| **AGENTS.md**    | **AI 主动读取时**  | **❌ 无**       |
| 工作流正文       | 触发时读取         | ❌ 无           |
| HANDOFF.md       | AI 主动读取时      | ❌ 无           |

#### 如何确保 AI 每次都读取

在 `task-management.md` 工作流中增加一步，让 AI 在每次开始任务时主动读取：

```markdown
### 1.0 读取项目规则
在开始工作前，先读取项目根目录的 `AGENTS.md` 文件，遵循其中的规则。
```

#### 与 User Rules 的对比

| 特性         | User Rules（IDE 设置）   | AGENTS.md（项目文件）      |
|--------------|--------------------------|----------------------------|
| 配置位置     | VS Code 设置             | 项目根目录                 |
| 作用范围     | 所有项目的所有对话       | 仅当前项目                 |
| 加载方式     | 每次对话自动注入         | AI 主动读取                |
| 适合内容     | 通用偏好（语言、风格）   | 项目专属（硬件、连接方式） |
| Token 开销   | 每次对话固定消耗         | 仅读取时消耗一次           |

---

### 1.5 知识库（Knowledge Items / KI）

KI 是系统从历史对话中**自动提炼**的知识条目，由单独的 Knowledge Subagent 生成和维护。

**工作方式：**
- 每次对话结束后，系统可能会分析对话内容，提炼有价值的知识
- 知识以 `metadata.json` + `artifacts/` 的形式存储在 `~/.gemini/antigravity/knowledge/` 目录下
- 每次新对话会携带**最近访问的 KI 摘要**（标题 + 简介 + 文件路径），不含完整内容
- AI 在需要时通过 `view_file` 读取完整的 KI artifact

**Token 开销：** 每个 KI 摘要约 **50-150 token**，只在相关时才会读取完整内容。

---

### 1.6 对话历史摘要

系统会保留最近 20 条对话的概要信息（Conversation ID、标题、时间、用户目标摘要）。

**Token 开销：** 每条对话摘要约 **100-200 token**，20 条共约 **2000-4000 token**。

---

### 1.7 Token 与文字的换算关系

Token 不等于单词或字符，取决于语言和分词器（tokenizer）。

**英文：**

| 单位 | 换算比例                               |
|------|----------------------------------------|
| 单词 | ~0.75 个单词 = 1 token（1 单词 ≈ 1.3 token） |
| 字符 | ~4 个字符 = 1 token                    |

```
"Hello world"     → 2 token
"SafeToAutoRun"   → 3 token（驼峰拆分）
"rm -rf"          → 3 token
```

**中文：**

| 单位   | 换算比例                     |
|--------|------------------------------|
| 汉字   | ~1-2 个汉字 = 1 token        |
| 字符   | ~1.5 个字符 = 1 token        |

```
"保存进度"         → ~3-4 token
"任务管理核心流程"   → ~5-6 token
```

**同义内容的 Token 对比：**

| 内容             | 中文          | 英文                   | 差异             |
|------------------|---------------|------------------------|------------------|
| "保存工作进度"   | ~4 token      | "Save progress" ~2 token | 中文多 ~100%    |
| 100 字说明       | ~60-70 token  | 同义英文 ~40-50 token  | 中文多 ~40-50%   |

**结论：同样的信息，英文比中文节省约 40-50% 的 token。** 这是工作流文件采用英文编写的原因。

## 三、对话的上下文机制

### 3.1 LLM 没有记忆

**LLM 本身没有记忆能力。** 每次你发一条消息，系统会把**整个对话历史重新拼接起来**，完整发送给 LLM。

### 3.2 同一对话中每条消息的 Token 消耗

**一个对话 = VS Code Gemini 聊天面板中的一个聊天线程**（对应一个唯一的 `conversation-id`）。

```
消息 1: 系统提示词(10,000) + 用户消息(50)                              = ~10,050
消息 2: 系统提示词(10,000) + 消息1历史(1,500) + 用户消息(30)           = ~11,530
消息 3: 系统提示词(10,000) + 消息1-2历史(3,000) + 用户消息(20)         = ~13,020
  ...
消息 30: 系统提示词(10,000) + 消息1-29历史(40,000) + 用户消息(30)      = ~50,030
```

**越聊越多，每条消息的 token 消耗线性增长。**当超过模型上下文窗口限制时，早期历史会被截断或压缩。

### 3.3 "已在上下文中"的含义

当 AI 在消息 1 中调用 `view_file` 读取了某个文件，该文件的内容会成为"消息 1 的工具调用结果"永久留在对话历史中。消息 2、3、... 发送时，LLM 可以直接看到这些历史内容，**不需要再次调用工具读取**。

```
同一对话中：
  消息 1 → AI 读了 task-management.md → 内容成为历史的一部分
  消息 2 → AI 看到历史中已有内容 → 不需要再读（但历史本身仍消耗 token）
  消息 3 → 同上
```

**注意：** "不需要重复读取"节省的是工具调用的开销，但历史内容本身仍然作为上下文被发送，持续消耗 token。

### 3.4 新对话 = 一切重来

当你点击 **New Chat**，就是一个新的 `conversation-id`，上下文完全清空：

- 之前读过的文件内容 → 消失
- 之前的讨论和决策 → 消失
- AI 需要重新读取所有相关文件

这就是 `/save-progress` + `/resume-work` 的核心价值：让你可以安全开新对话，同时通过 `HANDOFF.md` 保持任务进度。

## 四、长对话 vs 短对话的影响

### 4.1 场景对比

**❌ 一天都用同一个对话窗口：**
```
 9:00  任务A（修bug）    消息 1-10   历史累计 ~8,000 token
10:30  任务B（加功能）   消息 11-25  历史累计 ~20,000 token
14:00  任务C（改UI）     消息 26-40  历史累计 ~35,000 token
16:00  任务D（写文档）   消息 41-50  历史累计 ~45,000 token

→ 消息 50 的总 token：10,000(系统) + 45,000(历史) = 55,000 token
  其中任务A/B/C的历史约 35,000 token 是完全无用的！
```

**✅ 每个任务用新对话：**
```
 9:00  [对话1] 任务A    消息 1-10   历史 ~8,000 → 完成后开新对话
10:30  [对话2] 任务B    消息 1-15   历史 ~12,000 → 完成后开新对话
14:00  [对话3] 任务C    消息 1-15   历史 ~12,000 → 完成后开新对话
16:00  [对话4] 任务D    消息 1-10   历史 ~5,000

→ 消息的总 token：10,000(系统) + 5,000(历史) = 15,000 token
  上下文干净，只包含当前任务相关内容！
```

### 4.2 影响不仅是成本

| 影响         | 长对话（一个窗口做一天） | 短对话（按任务分）       |
|--------------|--------------------------|--------------------------|
| Token 消耗   | 越来越高                 | 保持低位                 |
| AI 响应速度  | 越来越慢                 | 始终快速                 |
| AI 准确度    | 下降（被无关历史干扰）   | 高（上下文干净）         |
| "遗忘"风险   | 高（早期内容被截断）     | 低                       |

## 五、提高效率的最佳实践

### 5.1 对话管理策略

```
核心原则：一个功能任务 = 一个对话窗口

完成功能 A → /save-progress → 开新对话 → /resume-work → 做功能 B
```

| 情况                             | 建议                                  |
|----------------------------------|---------------------------------------|
| 同一开发任务的连续工作           | 留在同一对话，避免重复加载            |
| 任务完成，开始新任务             | 开新对话，清除无关上下文              |
| 对话太长，AI 开始"遗忘"         | 开新对话，用 `/resume-work` 恢复      |
| 换了完全不同的话题               | 开新对话                              |
| 简单问题（"这个 API 怎么用？"）  | 开新对话，快问快答                    |

### 5.2 工作流优化

| 优化点                               | 做法                                 | 效果                     |
|--------------------------------------|--------------------------------------|--------------------------|
| 精简 description                     | 只写触发关键词，不写详细说明         | 每个工作流省 20-50 token |
| 控制数量                             | 只创建真正常用的工作流               | 避免索引膨胀             |
| 不要在 description 中重复正文内容    | description 负责匹配，正文负责执行   | 避免重复消耗             |
| 合并相似工作流                       | 功能相近的合并为一个                 | 减少索引条目             |

**优化前：**
```yaml
description: 保存当前工作进度，用于中断后恢复。当用户需要中断工作时，AI 会自动完成到安全点，更新 HANDOFF.md，并输出恢复指引。触发方式：用户发送 /save-progress 或 "保存进度"、"停止工作"、"我要走了"、"暂停" 等
```

**优化后：**
```yaml
description: 保存工作进度。触发词：/save-progress、保存进度、停止工作、暂停
```

### 5.3 消息效率

| 优化点               | 说明                                                                |
|----------------------|---------------------------------------------------------------------|
| **批量提需求**       | 一次说清楚所有要改的内容，避免多轮对话的重复上下文开销              |
| **提供精确路径**     | 告诉 AI 具体文件路径，减少搜索工具调用                              |
| **使用斜杠命令**     | `/save-progress` 比 "帮我保存一下当前进度" 更精确，减少匹配推理    |

### 5.4 User Rules（全局规则）

在 VS Code 中配置：`Settings → Gemini Code Assist → User Rules`

- 保持简洁，只写真正需要全局生效的规则
- 不要把项目特定的规则放在全局设置中（那些放 `AGENTS.md`）

## 六、HANDOFF.md — 跨对话的任务桥梁

`HANDOFF.md` 是持久化的任务交接文件，存在于项目根目录中，跨对话存活。

**与 AI 内部 task.md 的对比：**

| 特性     | task.md                                        | HANDOFF.md         |
|----------|------------------------------------------------|--------------------|
| 存储位置 | AI 系统目录（`~/.gemini/antigravity/brain/`）  | 项目根目录         |
| 生命周期 | 仅当前对话                                     | 跨对话持久化       |
| 可见性   | 仅 AI 可见                                     | 用户和 AI 都可见   |
| 用途     | 当前对话的任务追踪                             | 跨对话的任务交接   |

`HANDOFF.md` 配合 `/save-progress` 和 `/resume-work` 工作流，实现了"开新对话不丢进度"的核心能力。

└── z-readme/
    ├── AI-develope.md           ← 本文档
    └── AI开发工作中断与恢复方案.md
```

```
Token 消耗层次：

 固定消耗（每条消息都带）
  └─ 系统提示词 + 工具定义 + 工作流索引 + KI摘要 + 对话摘要  ~10,000-15,000
