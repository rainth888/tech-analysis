我看到了关键信号：你的笔记本 WSL 环境把大量 Windows 可执行文件注入到了 PATH，且正在 Windows 盘符（/mnt/g）上构建。这两点极易导致生成 CRLF 的脚本。

根因分析

- PATH 污染：日志中 PATH=.../mnt/c/.../GnuWin32/bin ... Git/cmd ... Python311 ... 等大量 Windows 路径被注入到 WSL。更明显的是在 WSL 里执行 wsl --version 返回的是 “Wsman Shell commandLine”，说明
Windows 上另一个名为 wsl 的可执行文件抢先匹配了。这种 PATH 污染会让构建链条里不经意调用到 Windows 程序（sed/awk/python/node 等），这些程序默认输出 CRLF。
- Windows 盘构建：笔记本的工具链路径在 /mnt/g/...，你大概率在 Windows NTFS 挂载目录下构建。即便工具是 Linux 的，只要某个环节调用了 Windows 程序或 Windows 编辑器改写，目标脚本就会变成 CRLF。
- Git 行尾策略不一致（高概率）：如果仓库在 Windows 下曾用 core.autocrlf=true 检出，脚本在 WSL 内就是 CRLF 开始的，构建再把这些脚本拷贝/拼接，最终产物仍是 CRLF。台式机那边多半没有这个问题。

立即修复（建议按优先级执行）

- 仅用 Linux 工具链：在笔记本 WSL 设置禁用 Windows PATH 注入。
    - 编辑 /etc/wsl.conf：
    [interop]
    appendWindowsPath=false
- 在 Windows PowerShell 执行：wsl --shutdown，重新进入 WSL。
- 校验：echo $PATH 应只含 /usr、/bin、/home 等，不应有 /mnt/c 路径。which sed awk python node sh 均应指向 WSL 内。
- 改为在 WSL 文件系统内构建：把 SDK/工程移动到 ~/work/...（ext4），不要在 /mnt/c|d|e|g 等 Windows 盘符下构建。例：
    - cp -a /mnt/g/ax630c_ipc_AX620E_SDK ~/sdk_ax630c
    - 在 ~/sdk_ax630c 里重新构建并验证生成脚本为 LF：file out/*.sh 或 dos2unix -ic out/*.sh
- 强制仓库脚本为 LF：添加 .gitattributes 规范并归一化行尾。
    - 在仓库根目录创建/补充：
    *.sh text eol=lf
    Makefile text eol=lf
    *.mk text eol=lf
- 归一化一次：git add --renormalize . && git commit -m "Normalize LF endings"
- 一次性清理产物行尾（临时补救）：在输出目录执行
    - find . -type f -name '*.sh' -print0 | xargs -0 dos2unix
    - 或用 sed -i 's/\r$//' file.sh

长期治理

- Git 配置统一：
    - 在 WSL：git config --global core.autocrlf input，git config --global core.eol lf
    - 在 Windows（如需同时编辑）：用 .gitattributes 强制 LF，Windows 编辑器默认保存为 LF。
- 固定 PATH：在 ~/.bashrc 或构建脚本最前面加一段，剔除 /mnt/ 路径，避免误用 Windows 工具：
    - export PATH=$(printf '%s' "$PATH" | tr ':' '\n' | grep -v '^/mnt/' | paste -sd:)
- 确保构建使用 WSL 的 python/node/sed/awk，不要从 /mnt/c/Program Files/... 调用。

- 有问题的脚本样本：file your_script.sh 是否显示 “CRLF line terminators”。
